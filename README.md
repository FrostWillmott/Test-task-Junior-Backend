# Test Assignment for Junior Backend Developer

### Запуск проекта (локально через Docker)

1. Скопируйте `.env.example` → `.env` и заполните значения:
   ```bash
   cp .env.example .env
   ```
2. Соберите и поднимите контейнеры:
   ```bash
   docker-compose up --build
   ```
   При старте автоматически применяются миграции, собирается статика и запускается сервер **Gunicorn** на `http://localhost:8000`.

### Production-ready изменения
Проект был доработан для соответствия стандартам production-окружения:
1. **Gunicorn**: Заменил стандартный `runserver` на производительный WSGI-сервер.
2. **WhiteNoise**: Настроена эффективная раздача статических файлов (CSS, JS, изображения админки) напрямую через Gunicorn.
3. **Multi-stage Docker build**: Образ разделен на этапы сборки и финального запуска, что уменьшает его размер и повышает безопасность.
4. **Безопасность**:
   - `DEBUG` теперь по умолчанию `False` в `docker-compose.yml`.
   - `ALLOWED_HOSTS` в режиме `DEBUG=False` требует явного указания в `.env`.
   - Статика собирается командой `collectstatic` при сборке образа.

### Тесты

Тесты находятся в папке `tests/`. Запуск тестов (в контейнере):
```bash
docker-compose run --rm web pytest -q
```

Также доступен отдельный `docker-compose.test.yml` для изолированного запуска тестов:
```bash
docker-compose -f docker-compose.test.yml up --build
```
### API
- POST `/api/sync/` — синхронизация всех медиа из Instagram в БД.
- GET `/api/posts/` — список постов (CursorPagination по умолчанию `page_size=10`).
- POST `/api/posts/{id}/comment/` — отправка комментария к посту по внутреннему ID.

### Документация API
После запуска проекта документация доступна в форматах:
- Swagger UI: `http://localhost:8000/api/schema/swagger-ui/`
- ReDoc: `http://localhost:8000/api/schema/redoc/`

Для ручного запуска проверок:
```bash

```

### Логирование
Настроено структурированное логирование. Логи выводятся в консоль контейнера `web`.
Уровень логирования для приложения `instagram` зависит от `DEBUG`: `DEBUG` в режиме разработки и `INFO` в продакшене.

Пример запроса на комментирование:
```bash
curl -X POST http://localhost:8000/api/posts/1/comment/ \
  -H 'Content-Type: application/json' \
  -d '{"text": "Hello IG!"}'
```

### Возможные улучшения

- **Асинхронная синхронизация**: Вынести `POST /api/sync/` в Celery-задачу, чтобы избежать таймаутов при большом количестве постов.
- **Rate limiting**: Добавить ограничение частоты запросов к Instagram API.
- **Retry-логика**: Автоматические повторные попытки при временных ошибках сети.
- **Защита sync-эндпоинта**: Добавить авторизацию для `POST /api/sync/` (например, кастомный permission с API Key из переменной окружения), чтобы предотвратить несанкционированные вызовы синхронизации.

## Задача
Разработать сервис на **Django + DRF**, который синхронизирует контент из Instagram в локальную базу данных и позволяет управлять комментариями через API.

## Стек
* **Python 3.10+**, **Django**, **DRF**.
* **PostgreSQL**.
* **Docker** & **Docker Compose**.

## Бизнес-требования

### 1. Синхронизация (Instagram -> DB)
**Эндпоинт:** `POST /api/sync/`
* Сервис должен выкачать **все** медиа-объекты пользователя из Instagram Graph API.
* Реализовать автоматический обход всех страниц пагинации (используя поле `next` в ответе API), пока не будет получен полный список постов.
* Данные сохраняются в локальную БД. Если пост уже был загружен ранее — обновить его поля актуальными данными из API (**Upsert логика**).

### 2. Список постов
**Эндпоинт:** `GET /api/posts/`
* Вывод списка всех сохраненных постов из локальной базы данных.
* Обязательна пагинация средствами DRF (`CursorPagination`).

### 3. Комментирование
**Эндпоинт:** `POST /api/posts/{id}/comment/`
* `{id}` в URL — это **внутренний Primary Key** поста в нашей базе данных.
* Система должна отправить текст комментария в Instagram API для соответствующего поста.
* В случае успеха — сохранить комментарий в локальную БД со связью к посту и вернуть сериализованные данные созданного комментария.

## Тестирование
Необходимо написать как минимум один интеграционный тест для эндпоинта создания комментария:
* Тест должен проверять создание записи в БД и корректность ответа API при успешном запросе.
* Тест должен проверять обработку ошибки, если пост с указанным `{id}` отсутствует в базе.
* Тест должен проверять обработку ошибки, если указанный пост существует в базе, но его нет уже в Instagram.
* **Требование:** Запрещено выполнять реальные HTTP-запросы к Instagram API во время тестов. Используйте инструменты мокирования (`unittest.mock`, `requests-mock` или аналоги) для имитации ответов от внешнего сервиса.

## Технические условия
* **Docker:** Весь проект (приложение + БД) должен подниматься одной командой: `docker-compose up --build`.
* **Архитектура:** Ожидается разделение ответственности. Логика запросов к внешнему API должна быть вынесена в отдельный слой (Services/Clients), а не находиться внутри Views.
* **Качество кода:** * Соблюдение PEP8.
    * Использование **Type Hinting** (аннотации типов).
    * Наличие кратких комментариев (**Docstrings**) к ключевым бизнес-методам (логика синхронизации, интеграция с API).
    * Обработка ошибок API.
